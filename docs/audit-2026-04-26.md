## 1. 策略层问题（最关键，影响选股有效性）

### 🔴 P0-1 龙虎榜 T 日盘后数据的时点风险（高）

- **位置**：[src/mo_stock/filters/lhb_filter.py:31-34](../src/mo_stock/filters/lhb_filter.py)、[src/mo_stock/storage/repo.py:157-160](../src/mo_stock/storage/repo.py)
- **问题**：`LhbFilter` 用 `trade_date` 直接等值查询当日龙虎榜，但**沪深交易所龙虎榜数据通常在 T 日 15:30 之后才发布**。代码中没有显式断言"调用时刻必须晚于 15:30"。
- **影响**：若运维误把调度任务提前（例如 14:00），会把"还没出"的龙虎榜当作可用信号，这其实是**用未来数据选当前持仓**——典型的 look-ahead bias。
- **建议**：
  1. 在 `daily_job.py` 入口处加时间断言：`if datetime.now(CN_TZ).time() < time(15, 30): raise RuntimeError(...)`。
  2. 或改为「选股目标 = T+1」时只取 `trade_date <= T-1` 的龙虎榜。
  3. 在 `lhb_filter.py` 的 docstring 顶部明确写出该假设。

### 🟠 P1-3 关键阈值未经历史回测验证（中）

- **位置**：[config/weights.yaml](../config/weights.yaml)、各 filter 内嵌阈值
- **问题**：
  - `limit`：首板 +20、连板 +30 的差异从何而来？
  - `moneyflow`：占比阈值 5% / 1% / 0.3% 为何这三档？业界更常见 3% / 1% / 0.5%。
  - `lhb`：净买入率 2% / 5% / 10%——10% 在 A 股极罕见，是否过严？
  - `sector`：TOP 5 板块（不是 TOP 3 或 TOP 10）的依据？
  - `docs/scoring.md` 给了**公式**，但没给**这些数字怎么来的**。
- **影响**：阈值若是凭经验拍的，等于"用未经验证的策略实盘"。在量化里这是大忌。
- **建议**：
  1. 在 `tests/integration/` 增加历史回放：拉 2024-Q4 ~ 2025-Q4 真实数据，用现有策略选股，统计 T+1 / T+3 胜率、夏普、最大回撤。
  2. 在 `docs/scoring.md` 补一节「阈值标定记录」，记录每个数字的来源（回测 IR / 业界惯例 / 暂时拍定）。
  3. 关键阈值做 ±10%、±20% 扰动测试，看排名是否稳定。

### 🟠 P1-5 龙虎榜未区分机构 vs 游资（中）

- **位置**：[src/mo_stock/filters/lhb_filter.py](../src/mo_stock/filters/lhb_filter.py)、[src/mo_stock/storage/models.py:227-229](../src/mo_stock/storage/models.py)
- **问题**：`Lhb.seat` JSONB 字段保留了席位信息，但 `LhbFilter` 全程不区分机构席位与游资营业部，统一按净买入率打分。两者信号截然不同：
  - **机构净买**：通常代表中长期看多，反包鲁棒性强
  - **游资净买**：短期题材，T+1 反向风险高
- **影响**：游资砸盘的票被误判为强势信号。
- **建议**：
  1. 在 ingest 阶段把 `top_inst`（机构）和 `top_list`（席位）分别落库，或在 `seat` JSONB 中记录类型标签。
  2. `LhbFilter` 给机构买入加权 ×1.2，游资 ×0.8。
  3. `detail` 字段中暴露 `inst_buy_amount` / `gambler_buy_amount` 便于追溯。

## 2. 数据接入层问题

### 🔴 P0-9 upsert 幂等性需实测验证（高）

- **位置**：[src/mo_stock/storage/repo.py:306-318](../src/mo_stock/storage/repo.py)、[src/mo_stock/storage/models.py:120-124](../src/mo_stock/storage/models.py)
- **问题**：使用 `pg_insert(...).on_conflict_do_update()`，但 `index_elements` 与表的复合主键约束名是否对齐没有显式验证。同一只股同一天**重跑两次**是否真的幂等，**没有专门的单测**。
- **影响**：若约束名不对齐，二次 ingest 会要么报错、要么写入重复行，报告里看到的是脏数据。
- **建议**：
  1. 写一个 SQLite + PG 双跑测试：同一份数据 ingest 两次，断言行数不变、字段被更新。
  2. 在 `repo.upsert_rows` 中显式断言 `set(conflict_cols) == set(model.__table__.primary_key.columns.keys())`。

### 🟠 P1-10 Tushare 重试捕获范围过宽（中）

- **位置**：[src/mo_stock/data_sources/tushare_client.py:110-115](../src/mo_stock/data_sources/tushare_client.py)
- **问题**：`@retry` 直接捕获 `Exception`，会把 Tushare 积分不足（403）、参数错误（4xx）这种"不可恢复"错误也重试 3 次。
- **影响**：浪费配额，且真正的失败信号被延迟。
- **建议**：限定为 `(requests.RequestException, ConnectionError, TimeoutError)`；4xx 直接抛 `TushareError`。

---

### 🟠 P1-11 空数据 vs 接口失败无法区分（中）

- **位置**：[src/mo_stock/data_sources/tushare_client.py:132-135](../src/mo_stock/data_sources/tushare_client.py)
- **问题**：周末调用返回 0 行 vs 接口超时返回 0 行，调用方都看到空 DataFrame，日志里也一样。
- **影响**：调度器可能误判"今天无数据，跳过"，错过补跑机会。
- **建议**：包装一个 `FetchResult` 数据类，含 `rows / source ('ok' | 'no_data' | 'api_error') / fetched_at`。

### 🟠 P1-12 `_parse_date` 时区与格式风险（中）

- **位置**：[src/mo_stock/ingest/ingest_daily.py:374-384](../src/mo_stock/ingest/ingest_daily.py)
- **问题**：`pd.to_datetime(s).date()` 不指定 format，pandas 会做模糊推断；若服务器时区非 UTC+8，date 对象可能差一天。
- **建议**：显式 `datetime.strptime(s, "%Y%m%d").date()`，并在文档明确「所有日期字段 = A 股交易日（UTC+8 视角的纯日期）」。

### 🟠 P1-13 SQLite vs PostgreSQL upsert 兼容（中）

- **位置**：[src/mo_stock/storage/repo.py:306](../src/mo_stock/storage/repo.py)
- **问题**：`pg_insert(...).on_conflict_do_update()` 是 PG 方言。集成测试若用 SQLite，upsert 会失败。
- **建议**：在 `_make_engine()` 后判断 `dialect.name`，分支调用 `pg_insert` 或 `sqlite_insert`。
